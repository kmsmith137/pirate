# GIL audit for pybind11 wrappers

## Background

pybind11's `py::call_guard<py::gil_scoped_release>()` releases the GIL during
the C++ function body, allowing other Python threads to run concurrently. The
sequence is:

1. Arguments converted (GIL held -- type_caster runs here)
2. GIL released
3. C++ function executes
4. GIL re-acquired
5. Return value converted (GIL held -- type_caster runs here)

**ksgpu::Array type_caster**: Both `load()` (Python->C++) and `cast()` (C++->Python)
call Python/numpy C API (`PyObject_GetAttrString`, `PyArray_New`, `PyImport_ImportModule`,
`Py_INCREF`, etc.) and **require the GIL**. However, `call_guard<gil_scoped_release>`
is safe to use with Array arguments/returns -- the GIL is only released during step 3.

**Lambdas that call pybind11 API**: A few wrappers use lambdas that call
`py::make_tuple()` or `py::object` methods. These need the GIL during the
lambda body, so `call_guard` cannot be used directly. To drop the GIL in
these cases, manually scope `py::gil_scoped_release` around just the C++ call.

These wrappers already have `py::call_guard<py::gil_scoped_release>()`:

## Should drop GIL: thread-backed start/stop/join

These methods spawn or join threads, wait on condition variables, or do
network I/O. Holding the GIL here can deadlock a multithreaded Python program.

| Wrapper | Method | Notes |
|---|---|---|
| `FakeXEngine` | `start` | Spawns worker threads, may block on barrier |
| `FakeXEngine` | `stop` (lambda) | Joins worker threads |
| `Receiver` | `start` | Spawns listener + reader threads |
| `Receiver` | `stop` (lambda) | Joins threads |
| `Receiver` | `get_metadata(blocking=True)` | Waits on condition variable |
| `FrbServer` | `start` | Starts receivers + gRPC service |
| `FrbServer` | `stop` (lambda) | Stops server, joins threads |
| `FileWriter` | `stop` (lambda) | Joins writer threads |
| `AssembledFrameAllocator` | `get_frame` | Blocks until a frame is available |
| `AssembledFrameAllocator` | `initialize` | Entry point for thread-backed class; may block |

For the lambda wrappers (`stop` methods), the lambdas are simple (`self.stop()`)
and don't use pybind11 API, so `call_guard<gil_scoped_release>` works directly.

## Should drop GIL: GPU synchronization

These methods enqueue CUDA stream events and may block waiting for GPU work.
They take only scalar/pointer args (no Array args), so `call_guard` is safe.

| Wrapper | Method | Notes |
|---|---|---|
| `GpuDedisperser` | `acquire_input` (lambda) | GPU stream sync; lambda only does reinterpret_cast |
| `GpuDedisperser` | `release_input` (lambda) | GPU stream sync |
| `GpuDedisperser` | `acquire_output` (lambda) | GPU stream sync |
| `GpuDedisperser` | `release_output` (lambda) | GPU stream sync |
| `GpuDedisperser` | `allocate` | Allocates GPU + host memory (CUDA malloc) |
